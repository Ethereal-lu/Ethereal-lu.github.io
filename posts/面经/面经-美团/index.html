3、如果有 100 个元素，且之后不会增加，存储在 hashmap 中，初始容量应该设置为多少？

答：64？ 100？ 128？ 瞎蒙

正确：为了不触发扩容操作，初始值应该设置为 129 到 256 的任意一个值；因为在 hashmap 的构造函数中，会将输入的初始容量转换为第一个大于等于它的 2 的幂次值并赋值给 threshold ，而在 put 操作中，会将该 threshold 赋值给 capacity ，并根据 capacity * loadfactory得到新 threshold 。因此，如果初始值设置为小于等于 128，则经过 put 操作后，threshold 会变为 96，小于 100，如此一定会触发扩容操作。故为了不触发扩容需保证初始容量为 256。

4、如果有 41 个 int 数字，另外再给一个数字，如何判断这个数字在不在之前的 41 中？（后续准备八股文才反应过来，应该是40亿吧）

答：hashset 或 布隆过滤器，瞎蒙。布隆过滤器存在误差。

应该使用位图。由于 2^32 大约42亿多，故可以申请 2^32 个 bit，每个数对应一个bit，若有值该bit位为1，否则为0；如若 1 在，则第一个 bit 位置为 1，若17在，则第17个 bit 位置为 1。这样判断一个值在不在时只需判断该 bit 位是否为 1 即可。40 亿个 32 位int 大约 16GB，而使用位图算法，大约只需 500MB，可以在内存中判断。

  • 此处补充一问：如果是 40 亿个字符串，如何判断给定字符串是否在这 40 亿个中？

    • 字符串到位图需要使用哈希映射，会产生冲突，因此不能使用位图。应该使用字典树。

  • 从10亿个数据中找到第5亿个怎么找？

    • 将数据按顺序划分到 2^16个槽中，10亿数据不到2^32个，可以给每个槽放2^16个数据，这样第5亿个数据在哪个槽就知道了，然后将该槽的数据全部加载到内存扫描即可。

7、什么时候会触发 full GC？（ full GC 和 Major GC 是等价的）

  • 调用System.gc()：只是建议虚拟机执行 Full GC，但是虚拟机不一定真正去执行。
  • 老年代空间不足：老年代空间不足的常见场景比如大对象、大数组直接进入老年代、长期存活的对象进入老年代等。
  • 空间分配担保失败

8、 full GC 对服务器有什么影响？

因为 full GC 需要所有的业务线程都进入安全点，所以这些线程就不能执行任何字节码指令了，只有出了安全点才能继续执行，所以会发生 STW，导致不能正常提供服务。

9、如果服务器在运行期将发生了 full GC ，你会如何处理？

  • 增大 survivor space ；可避免在 Minor GC 时由于 survivor space 太小而晋升到老年代。
  • 增大老年代空间；降低 full GC 的频率。
  • 代码层面避免使用大数组和大对象。

尽量做到让对象在Minor GC阶段被回收、让对象在新生代多存活一段时间及不要创建过大的对象及数组。

频繁发生full gc如何处理？

  1. 观察年轻代 gc的情况，如果每次gc后存活的对象都大于survivor区就调整内存分配比例
  2. 观察老年代的内存情况，如果内存持续上涨可能是大对象造成内存溢出
  3. 如果年轻代和老年代内存都较低，可能是元数据区加载太多东西
  4. 排查看代码中是否有循环System.gc()

10、执行一次 System.gc() 的流程？ 不会！扯了一通可达性分析

深入理解System.gc()

System.gc()实际上调用的是RunTime.getRunTime().gc()，这个方法是一个 native 方法：首先，根据 DisableExplicitGC 这个 JVM 启动参数的状态，确定是否会 GC，如果需要 GC，不同 GC 会有不同的处理。 以 G1 GC 为例：如果是 System.gc() 触发的 GC，G1 GC 会根据 ExplicitGCInvokesConcurrent 这个 JVM 参数决定是YoungGC 还是 FullGC。

System.gc() 相关的 JVM 参数

  • DisableExplicitGC： 是否禁用显式 GC；默认是不禁用的。
  • ExplicitGCInvokesConcurrent： 执行轻量并行 GC （YoungGC）还是 FullGC；默认执行 FullGC。

12、进程间通信方式

每个进程各自有不同的用户地址空间,任何一个进程的全局变量在另一个进程中都看不到，所以进程之间要交换数据必须通过内核,在内核中开辟一块缓冲区,进程A把数据从用户空间拷到内核缓冲区,进程B再从内核缓冲区把数据读走,内核提供的这种机制称为进程间通信。

  • 管道通信：管道是一种半双工的通信方式，数据只能单向流动，而且只能在具有亲缘关系的进程间使用。进程的亲缘关系通常是指父子进程关系。
  • 命名管道FIFO：有名管道也是半双工的通信方式，但是它允许无亲缘关系进程间的通信。
  • 消息队列通信：消息队列，是消息的链接表，存放在内核中。一个消息队列由一个标识符（即队列ID）来标识。
  • 信号量通信：信号量是一个计数器，用于实现进程间的互斥与同步，而不是用于存储进程间通信数据。
  • 信号： 信号是在软件层次上对中断机制的一种模拟，是一种异步通信方式，信号可以在用户空间进程和内核之间直接交互。
  • 共享内存通信：共享内存就是映射一段能被其他进程所访问的内存，这段共享内存由一个进程创建，但多个进程都可以访问。共享内存是最快的 IPC 方式，它是针对其他进程间通信方式运行效率低而专门设计的。
  • 套接字通信：可用于不同机器间的进程通信。

bean 的初始化过程

首先确定单例 bean 是在spring启动的时候初始化的，多例或懒加载是在 getBean 获取时初始化的。当spring容器开始加载时，首先会解析 AppConfig 类，然后根据 AppConfig 中指定的扫描路径找到加了四大注解的类并解析，类被解析后被构建为 BeanDefinition 类，其中存储了类的所有基本信息包括类型、父类名字、作用域等。最后遍历所有的 BeanDefinition ，将单例且不是懒加载的类进行初始化并放入单例注册表中。

RabbitMQ 数据丢失的场景

  1. 生产环境网络波动导致丢包
  2. 使用Confirm模式，接收端接收失败发送消息给生产者，生产者接收失败或未作处理
  3. 消息发送到exchange,但是queue和路由没有绑定
  4. 消息未做持久化处理，或者持久化不完全（只持久化Message，未持久化Queue，Exchange）
  5. 未做消息队列集群，消息会持久化到硬盘，但磁盘损坏，不能存储数据
  6. 普通集群节点消息同步的时候绑定Queue的节点挂了，需要等节点回复之后才能正常接收消息
  7. 消费端还没来得及确认就挂了，但消息未做处理
